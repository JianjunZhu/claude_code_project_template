# RESEARCH_RULES.md — 科研纪律详规

> 本文件是科研专章，比 CLAUDE.md 中的总则更细。它规定 Agent 与研究者在本项目中处理证据、数据、指标、benchmark、论文与基金时必须遵守的纪律。
> 文档分类提醒：本文件是**科研行为/证据纪律**，不记录具体项目事实（见 `docs/records/PROJECT.md`），不记录长期经验（见根目录 `MEMORY.md`），不记录逐次实验流水（见 `docs/records/EXPERIMENT_LOG.md`）。
>
> 本模板于 2026-06-19 创建。除该创建日期外，本文件不包含任何具体项目事实、实验结果或指标数值。
>
> 使用方式：下列条目均为可勾选清单（`[ ]` 未满足 / `[x]` 已满足）。每次涉及证据、结论、数据、提交前，对照相关小节逐条核对。

---

## 0. 统一证据等级（全模板共用同一套，低 → 高）

所有文档（本文件、`docs/records/EVIDENCE.md`、`docs/records/EXPERIMENT_LOG.md`、`docs/records/PROJECT.md`、`docs/records/RESULT_AUDIT.md`、`docs/records/PAPER_NOTES.md`）引用的证据等级**必须**指向以下同一套定义，不得各自另立标准：

| 等级 | 名称 | 含义 |
| --- | --- | --- |
| 1 | 假设 | 仅为想法/猜测，无任何实现或数据支持。 |
| 2 | 设计方案 | 有方法/协议设计，但尚未实现或运行。 |
| 3 | 实现正确性检查 | 代码可运行、单元/形状/数值健全性检查通过，尚无任务指标。 |
| 4 | 开发集验证 | 在开发集（dev）上得到初步指标，用于迭代，易过拟合。 |
| 5 | 内部验证 | 在 validation 集上的较稳定结果，仍可能反复调参。 |
| 6 | 冻结内部测试 | 协议/模型/阈值冻结后，在内部 test 集上一次性评估。 |
| 7 | 锁定 holdout 测试 | 在锁定、从未用于调参的 holdout 集上一次性评估。 |
| 8 | 外部验证 | 在外部、独立来源数据集上验证。 |
| 9 | 独立复现 | 由独立团队/独立环境复现同一结论。 |

**核心约束**

- [ ] 不得把低等级证据描述成高等级（例如把等级 4 的 dev 指标写成"测试集结果"或"已验证"）。
- [ ] 每条结论标注其当前证据等级，并说明达到更高等级还缺什么。
- [ ] 证据等级只能由真实 artifact 提升，不能由措辞提升。

## 1. 不得编造规则（最高优先级）

无真实 artifact 支持的内容，一律标 `PENDING` / `NOT VERIFIED` / `TBD` / `INCOMPLETE`，**不得**写成既成事实。

禁止在缺乏真实 artifact 时凭空给出：

- [ ] 实验结果、指标数值、置信区间、显著性。
- [ ] 数据规模、样本数、类别分布、划分比例。
- [ ] 对比方法 / baseline 的数值与结论。
- [ ] 训练状态（是否收敛、训练了多少 epoch/step）。
- [ ] 模型行为（鲁棒、泛化、对某类样本更好）。
- [ ] 论文结论、临床结论、可用性结论。
- [ ] 消融实验结果。
- [ ] 运行时间、吞吐、显存/内存占用、硬件规格。

判定口径：

- [ ] 信息未知 → `TBD`。
- [ ] 结果尚未运行或未验证 → `PENDING`。
- [ ] 结论存在但未被证据确认 → `NOT VERIFIED`。
- [ ] 实验/流程未跑完 → `INCOMPLETE`。
- [ ] 不确定时，宁可标占位符，不可猜测真实值。

## 2. 证据可追溯规则

每一个数字、每一条关键结论，都要能回答下面 6 个问题；缺任一项即视为**不可追溯**，需标注 `NOT VERIFIED`：

1. [ ] **结论是什么**：用一句话陈述。
2. [ ] **由哪个 artifact 支持**：具体文件/记录（如指标 json、日志、图、表）。
3. [ ] **artifact 在哪**：路径或归档位置（数据/权重/大输出走 manifest+checksum，不入 git）。
4. [ ] **何时生成**：日期/时间或 run id。
5. [ ] **由什么命令生成**：可复现的命令行（含脚本、配置、参数）。
6. [ ] **是初步还是最终证据**：对应第 0 节的证据等级。

可复现最小记录集（每条关键结论尽量齐备）：

- [ ] commit hash
- [ ] 运行环境（`<Python解释器>` / `<Conda环境>` / 依赖版本）
- [ ] seed
- [ ] 数据划分（train/validation/test、holdout、外部集）
- [ ] 配置文件与命令行参数
- [ ] checkpoint 路径 / `<模型权重路径>`
- [ ] 评估脚本
- [ ] 输出 manifest（含 checksum）

---

## 3. 结论表达规则

- [ ] 结论与证据等级绑定：先有等级，再有措辞。
- [ ] 区分"在 dev/validation 上观察到" 与 "已验证"：前者属等级 4–5，不得写成结论。
- [ ] 趋势 ≠ 结论：单次运行、未控随机性的差异只能写为"初步观察（PENDING）"。
- [ ] 比较类陈述（A 优于 B）必须给出：数据集、划分、指标、是否同一评估脚本、随机性处理。
- [ ] 任何"更好/提升 X"都需指明对照对象、指标口径（case-level / aggregate）与证据等级。
- [ ] 不得用绝对化措辞（最佳、SOTA、已解决、可临床使用）替代证据（见第 13 节）。

---

## 4. 负结果保留规则

负结果是科研资产，**只能归档，不能删除**。

- [ ] 失败的运行（崩溃、不收敛、NaN）记录并归档到 `experiments/records_archive/`。
- [ ] 被否定的假设保留，并注明否定它的证据。
- [ ] 数据泄漏风险、可疑的"过好"结果保留并标注。
- [ ] 指标前后不一致、复现失败保留并说明差异。
- [ ] 归档项保留可追溯信息（第 2 节六问），便于日后回溯。
- [ ] 删除任何记录/日志/指标/草稿/配置属不可逆操作，需明确指令，且优先归档（见第 14 节）。

---

## 5. 冻结测试纪律（防止信息泄漏与过拟合测试）

针对 holdout（等级 7）与外部测试集（等级 8）：

- [ ] **不**在 holdout/外部测试集上调 checkpoint、阈值、架构、loss、超参。
- [ ] **不**反复使用 holdout/外部集来"挑"方法或模型选择。
- [ ] 测试前先冻结：评估协议、评估脚本、模型权重、阈值、配置。
- [ ] 冻结点记录 commit hash、配置快照、模型 checkpoint 路径与 checksum。
- [ ] 每个锁定测试集**一次性**评估；如必须复评，记录原因并视作新的、降级的证据。
- [ ] 任何疑似泄漏（划分污染、特征穿越、标签泄漏、预处理统计跨集泄漏）**必须**记录在案，不得掩盖。
- [ ] 调参/模型选择只允许在 dev/validation（等级 4–5）上进行。

## 6. 数据集规则

数据/权重/大输出默认**不入 git**，以 manifest + checksum 替代。空模板下具体值用占位符（`<数据集名称>`、`<数据路径>`、`TBD`）。每个数据集应记录：

- [ ] 数据集名称 `<数据集名称>` 与版本号。
- [ ] 来源 / 提供方 / 许可与使用授权范围。
- [ ] 存放路径 `<数据路径>`（隐私路径不入 git）。
- [ ] 样本数与基本统计（`TBD`，未知不猜）。
- [ ] 类别 / 标签定义与分布（`TBD`）。
- [ ] 划分方式：train / validation / test / holdout / 外部集，及划分依据（按病例/受试者/中心避免泄漏）。
- [ ] 划分的随机性控制（seed）与划分清单（id 列表）。
- [ ] 预处理 / 清洗 / 归一化流程及其统计量的计算范围（仅用训练集统计，防泄漏）。
- [ ] 排除 / 纳入标准与被排除样本记录。
- [ ] 标注协议、标注者、标注一致性（如适用，`TBD`）。
- [ ] 数据 manifest 文件路径与 checksum。
- [ ] 隐私 / 合规要求（去标识化、可分享范围）。
- [ ] 已知问题（噪声、缺失、偏倚、潜在泄漏）。

## 7. 指标规则

- [ ] 明确**主要指标** `<主要指标>`：决定结论的核心指标，事先确定，不事后改。
- [ ] 明确**辅助指标** `<辅助指标>`：辅助理解，不单独支撑主结论。
- [ ] 记录**失败 / 退化指标**：用于发现坏样本、崩溃、退化模式，必须报告而非隐藏。
- [ ] 记录**安全 / 风险指标**：与误诊、漏检、风险后果相关的指标（如适用）。
- [ ] 区分 **case-level**（逐样本/逐病例）与 **aggregate**（整体汇总）：报告时注明口径，避免用 aggregate 掩盖 case-level 失败。
- [ ] 指标计算由**固定评估脚本**产出，脚本纳入版本管理并记录 commit hash。
- [ ] 报告指标时附带：数据集、划分、证据等级、随机性处理、生成命令。
- [ ] 指标变化必须可由 artifact 解释，不可仅凭印象描述"提升"。

## 8. benchmark 规则

- [ ] **公平划分**：所有方法使用同一划分，不为某方法定制有利划分。
- [ ] **防泄漏**：按病例/受试者/来源划分；预处理统计仅来自训练集；检查重复样本跨集污染。
- [ ] **baseline 可复现**：记录 baseline 来源、版本、commit hash、配置、命令；无法复现的 baseline 标 `NOT VERIFIED`，不得直接引用其数值当作己方对比结论。
- [ ] **统一预处理**：所有方法共享同一预处理与数据加载流程，差异需显式记录。
- [ ] **固定评估脚本**：所有方法用同一评估脚本与同一指标实现。
- [ ] **明确 leaderboard 规则**：排名指标、并列处理、提交次数限制、是否允许测试集多次评估，事先写清。
- [ ] **版本管理**：benchmark 数据版本、划分版本、评估脚本版本均加版本号并记录。
- [ ] **排除标准**：明确哪些样本/方法被排除及原因，排除清单可追溯。
- [ ] benchmark 结论同样遵守证据等级，不得把开发期内部对比夸大为最终结论。

## 9. 论文写作规则

- [ ] 论文中每个数字、图、表都能映射到 `docs/records/EVIDENCE.md` / `docs/records/EXPERIMENT_LOG.md` 的具体 artifact。
- [ ] 论文结论的证据等级与正文一致，避免"标题党"强于证据。
- [ ] 方法描述与实际运行的脚本/配置一致（commit hash 可追溯）。
- [ ] 报告随机性处理（seed、重复次数、方差/区间），不以单次运行下结论。
- [ ] 负结果、局限性、失败模式在论文中如实呈现。
- [ ] 数据与代码可用性声明真实（仅承诺可公开的部分）。
- [ ] 未完成内容用 `INCOMPLETE` / `PENDING` 标注于草稿，不写成已完成。
- [ ] 不在论文中出现未经证据支持的临床/泛化/SOTA 主张（见第 13 节）。
- [ ] 写作记录归于 `docs/records/PAPER_NOTES.md`，与项目事实、证据分离。

## 10. 基金写作规则

明确区分时间状态与证据状态，**不得**把"拟开展"写成"已完成"：

- [ ] **已完成**：有真实 artifact 支持，标明证据等级。
- [ ] **在研**：正在进行，标注当前进度与 `INCOMPLETE`。
- [ ] **拟开展**：计划但未开始，明确标注为计划。
- [ ] **预期结果**：作为目标/假设陈述（等级 1–2），措辞为"预期/有望"，不写成既得结果。
- [ ] **风险**：列出技术/数据/可行性风险。
- [ ] **对策**：针对每条风险给出缓解方案与备选路径。
- [ ] 引用的前期工作数据需可追溯；他人成果标注来源。
- [ ] 基金占位字段用 `<基金项目类型>` 等，未知用 `TBD`。

## 11. 科研安全表达规则

证据不足时，**不得**作出下列声称：

- [ ] 临床有效 / 可用于临床 / 诊断更优。
- [ ] 泛化强 / 鲁棒 / 适用于未见分布。
- [ ] state-of-the-art / 超越所有现有方法。
- [ ] 已解决某问题 / 完全可靠。

应采用的谨慎措辞（按证据等级匹配）：

- [ ] "我们假设 / 我们预期"（等级 1–2）。
- [ ] "初步观察 / 仍需验证 / 当前证据仅支持……"（等级 4–6）。
- [ ] "在 holdout / 外部集上观察到，需独立复现确认"（等级 7–8）。
- [ ] 任何强结论须等级 7 以上且可追溯；临床类主张需对应更高等级与外部/独立证据，并明确尚未满足时标 `NOT VERIFIED`。

## 12. 可复现规则

每个可发布结论应附最小复现集（与第 2 节一致）：

- [ ] commit hash、环境、seed、数据划分、配置、命令行参数、checkpoint 路径、评估脚本、输出 manifest（含 checksum）。
- [ ] 复现失败视为负结果，按第 4 节归档并记录差异。

## 13. git 与提交规则

- [ ] 源码与轻量文档入 git。
- [ ] 数据、模型权重、大输出、概率图、医学影像**默认不入 git**，用 manifest + checksum 替代。
- [ ] 不提交密钥、token、隐私路径、受限数据。
- [ ] 提交信息可追溯到对应实验记录 / 结论。
- [ ] `.gitignore` 覆盖大体积与敏感产物。

## 14. 不可逆操作规则

- [ ] 删除数据 / checkpoint / 日志 / 指标 / 草稿 / 配置属不可逆操作。
- [ ] 仅在收到**明确指令**时执行，且优先**归档**而非删除（移入 `experiments/records_archive/` 或独立归档区）。
- [ ] 执行前确认有备份或归档副本；执行后记录操作原因与时间。

## 15. 执行前验证与运行监控

> 执行 / 工程纪律要点，完整操作规范见 `CLAUDE.md` 第 14 节；此处只列与证据、正确性相关的核对项。

- [ ] **小批量先行**：开启大批量 / 全量运行前，先在小批量（少量样本 / step / 单 case）上完全跑通，达到证据等级 3（实现正确性检查）后再放量。
- [ ] 明确"小批量跑通 ≠ 方法有效"：仅证明实现正确，不据此提升证据等级或下科研结论。
- [ ] **运行监控**：长任务设哨兵定时巡检（存活、`NaN` / 发散、资源、checkpoint 产出），异常及时止损并按第 4 节归档为负结果。
- [ ] **效率不换正确性**：为提速引入的近似 / 降精度 / 子采样须记录，并确认不影响结论；可能影响时标 `NOT VERIFIED`。
- [ ] 效率优化不得违反第 5 节冻结测试纪律或引入数据泄漏；不得为省时间省略第 2 / 12 节的可复现要素。
- [ ] 运行资源指标（吞吐、wall-clock、显存）记入 `docs/records/EXPERIMENT_LOG.md`，无真实运行前为 `PENDING`。

## 16. 科研攻关执行教训（来自真实攻关会话的人工校正）

> 以下教训源自真实科研攻关中**被反复证伪的判断**，凝练为执行纪律。它们集中针对最易翻车的几个环节：下重大结论、调超参、定研究方向、用算力。核心信条：**方法学 artifact 会两头骗人——负结论和正信号同样可能是方法学错误造成的，必须同等严格地排查。**

### 16.1 负结论与正信号同等严格验证

下任何重大结论（尤其**降级 / 撤 claim / 宣称"创新无效"**）之前，先穷尽排查方法学，再下结论：

- [ ] **先查训练-评测一致性**：评测的门控强度 / 掩码分布 / 预处理口径是否与训练匹配；不匹配会同时制造假负与假正。
- [ ] **警惕伪重复虚高显著性**：只重采单一噪声源（如同一数据只换一个随机种子的某一处）不是独立重复，会虚高显著性；统计检验须对**真正独立**的重复采样（与第 7 节指标、`CLAUDE.md` 第 8 节 $N \ge 7$ 一致）。
- [ ] **加"学到的量 vs 平凡基线"关键控制**：用对照（如 closed vs plain＋原始坐标）排除"平凡效应冒充真增益"——位置 / 形状等平凡线索可能让预测器"看起来"学到了东西。
- [ ] **正信号同样查方法学**：好结果先排除"配置 bug / 数据泄漏 / 平凡捷径"，再归功于创新；不得只对坏结果做尽职调查、对好结果放行。

### 16.2 先测 null/base 假设，再把问题归因给创新

- [ ] 出现异常（崩坏 / 不收敛 / 异常增益）时，**先跑"关掉创新的纯 baseline"对照**，定位问题归属。
- [ ] 若纯 baseline 在该配置下**本身就崩** → 问题在**基础训练配置**（如大 batch 配过高 LR 把 base 训崩），不在创新；**未做此对照前不得把锅甩给创新机制**。

### 16.3 盲调超参前，先调研 prior art（论文＋官方代码）

- [ ] 卡在收敛 / 稳定性时，**先查同领域已有工作**（论文＋官方代码）怎么设 batch / LR / warmup / mask / momentum / 防坍缩技巧，从中找答案。
- [ ] 网格盲搜是**末选不是首选**：prior art 的成熟设定优先级高于凭感觉试参；调研后仍无解再做受控搜索。

### 16.4 重大研究方向变更不由 agent 单方过早拍板

- [ ] 降级 / 撤创新 / 换路线等**重大决策**，须把**完整证据＋不确定性**摊给用户（PI）判断，**不替其拍板**（与 `CLAUDE.md` 第 15 节"可被用户降档"、第 16 节汇报一致）。
- [ ] **dev 小尺度负结果不足以撤创新**：小尺度结论有局限，必要时**扩到大尺度**暴露其局限后再下结论，不以小尺度负结果过早否定。

### 16.5 算力：吃满但不超占

- [ ] **只占需要的卡数**，但把已占用卡的**核心利用率与显存尽量填满**（避免 batch 过小只占 10/80GB 浪费，与 `CLAUDE.md` 第 14.1 节"不让计算单元饥饿"一致）。
- [ ] **已被对照证否的实验及时停掉**，把算力让给仍有效的实验。

### 16.6 被定向时持续迭代、不轻易收尾

- [ ] 负结论只有在**真正穷尽设计空间**（不同策略 / 超参 / 设计变体）后才可信；半途的负结论按 `PENDING` / `INCOMPLETE` 处理。
- [ ] 在用户给定方向下**持续迭代**，不因一两次失败就收尾或改判（与默认 ultracode 强度一致，见 `CLAUDE.md` 第 15 节）。
