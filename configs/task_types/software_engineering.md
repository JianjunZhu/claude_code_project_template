# 任务类型配置：software_engineering（软件工程）

> 配置可选，基础模板不依赖任何单一任务类型。
> 本文件仅在当前项目确属"软件工程"任务时启用；启用方式为在 `docs/PROJECT.md` 标注任务类型并按需引用本文件。所有具体取值用占位符表达，未知=TBD、未验证=PENDING、未确认=NOT VERIFIED、未完成=INCOMPLETE。

## 1. 任务目标

- 研究目标：`<研究目标>`（TBD）。
- 任务形式：对 `<项目名称>` 的代码/工具/管线进行实现、修复或重构，核心原则是"先查看后编辑、做小而可验证的修改"。
- 任务边界：明确改动范围、受影响模块与回归风险（TBD）；不可逆操作需明确指令。
- 评价口径：以是否可验证（测试/复现）、是否最小化改动、是否引入回归为口径。
- 未验证改动效果标 PENDING，未确认行为标 NOT VERIFIED。

## 2. 默认交付物

- 改动代码与 diff（小步、聚焦单一目的）。
- 测试：新增/更新的单元或集成测试（路径 TBD）。
- 变更说明：改动动机、范围、验证方式（记录 commit hash）。
- 复现命令：`<主脚本>` 或脚本入口与参数（TBD）。
- 必要时更新文档（README、配置说明）。
- 实验记录：`experiments/records/` 中与科研结果相关的运行（如适用）。

## 3. 推荐指标

> 取值留占位，不得填未经真实运行支持的数字。

- 正确性：测试通过情况、回归是否引入（PENDING，需真实运行）。
- 改动规模：改动行数/受影响文件数（越小越易验证）。
- 覆盖：关键路径是否有测试覆盖（`<辅助指标>`=TBD）。
- 可复现性：能否由记录命令一键复现行为。
- 性能（如相关）：基准前后对比（取值 TBD/PENDING，需真实测量）。

## 4. 常见失败模式

- 未先查看就编辑：忽略既有约定/调用关系，造成隐性破坏。
- 改动过大：一次混入多个目的，难以验证与回退。
- 缺测试：改动效果声称为已验证（NOT VERIFIED）却无运行支持。
- 不可逆操作未确认：删除数据/配置/日志前未取得明确指令。
- 把未运行的预期写成既成结果（应标 PENDING）。
- 提交密钥/token/隐私路径或将大体积产物入 git。

## 5. 证据要求

引用统一证据等级（1→9）；软件正确性多对应等级 3（实现正确性检查）。

- 每项"已修复/已实现"结论须可回答：是什么 / 哪个 artifact（测试/运行日志）/ 在哪 / 何时 / 何命令 / 等级是几。
- 改动效果未经真实运行验证前标 PENDING，行为未确认标 NOT VERIFIED。
- 先查看相关代码与调用关系，再做最小改动；优先归档而非删除。
- 不可逆操作需明确指令；源码与轻文档入 git，数据/权重/大输出用 manifest+checksum。
- 不提交密钥/token/隐私路径。
- 不得声称"已修复/性能更优"，除非有测试或基准支持。

## 6. 优先查看的文件

1. `CLAUDE.md`（Agent 行为规则与编辑纪律，最优先）。
2. `docs/PROJECT.md` 与 `docs/TASK_BRIEF.md`（项目事实与范围）。
3. 相关源码与既有测试（先查看调用关系再改）。
4. `docs/RESEARCH_RULES.md`（不可逆操作与 git 纪律）。
5. `scripts/` 与 `configs/`（入口与配置约定）。
6. `docs/EXPERIMENT_LOG.md`（若改动影响科研结果复现）。

## 7. 典型输出

- 聚焦单一目的的小步 diff 与变更说明（含 commit hash）。
- 新增/更新的测试与运行日志（状态 PENDING 直至真实运行）。
- 复现命令与参数记录。
- 必要的文档更新（相对路径引用）。
- 如影响科研结果：评估运行 manifest（环境、seed、命令、时间）与实验记录条目。
